## Long-term Memory for AI Agents
### 1.为什么需要外部记忆
LLM 本身无状态。每次调用都是全新的，模型不保留任何上下文。

Agent 的"记忆"= 开发者在每次调用前，把相关历史手动注入 prompt。

就像每次和一个失忆的人对话——你必须在开场白里告诉他"上次我们聊了什么、你是谁、你的目标是什么"，他才能接着干活。没有外部记忆系统，Agent 只能处理单轮任务，无法支撑跨会话的连续工作流。

### 2.四种记忆存储形态

| 类型 | 实现方式 | 适合场景 | 局限 |
|---|---|---|---|
| In-context | 直接拼进 prompt | 短期对话历史 | token 上限，成本高 |
| 向量数据库 | 语义 embedding + 检索 | 大量非结构化历史 | 召回精度依赖 embedding 质量 |
| 结构化数据库 | SQL / KV store | 用户偏好、事实信息 | 不适合模糊查询 |
| Agent 自写摘要 | 每轮结束让 Agent 压缩总结 | 长期会话压缩 | 摘要可能丢失细节 |

#### 应用场景
通常不是单选——生产级 Agent 会组合使用：
- 短期用 in-context
- 长期用向量库语义召回
- 用户画像用结构化存储

### 3.长期一致性的真正挑战
#### 不是"存得越多越好"

记忆注入过多会导致：
- Context 被旧信息稀释，当前任务注意力下降
- 过时信息干扰判断（用户偏好已经改变但旧记录仍在召回）
- Token 成本线性增长

#### 三个核心设计问题

1. **何时召回**：每次都召回 vs 判断相关性再召回
2. **召回多少**：Top-K 相关片段，K 怎么定
3. **记忆有效期**：如何标记/删除过时记忆（用户换工作了，旧的职业信息还要保留吗）

#### 当前主流方案

- **MemGPT / Letta**：Agent 自主管理内外存储，决定什么进 context、什么存档
- **Mem0**：专门做 Agent 记忆层的开源库，自动提取、更新、召回记忆
- **LangGraph + checkpointer**：图结构工作流 + 状态持久化

记忆系统的难点不是存储，是检索策略和遗忘策略。

## EOA vs 智能账户 vs 多签账户：权限与安全边界对比

### 1. 谁持有控制权

| 账户类型 | 控制权归属 |
|---|---|
| EOA | 单一私钥持有者，私钥即一切 |
| 智能账户 | 合约逻辑定义控制权，可设置多个 owner 或恢复人 |
| 多签账户 | M-of-N 密钥组，控制权分散在多个持有者之间 |

EOA 是最集中的控制结构——一把钥匙开一把锁，没有任何冗余。

### 2. 谁可以发起交易

**EOA**：只有私钥持有者本人。没有私钥，任何人都无法代为操作。

**智能账户**：由合约规则决定。可以设置单一 owner 发起，也可以允许授权的 session key（临时密钥）在限定范围内发起交易，甚至支持 Relayer 代付 Gas 后替你广播。

**多签账户**：任何一个签名方都可以"发起提案"，但提案本身不生效——必须达到阈值（如 2-of-3）后才真正执行。发起和批准是两个分离的动作。

### 3. 是否支持多人确认

| 账户类型 | 多人确认 |
|---|---|
| EOA | 不支持，单点决策 |
| 智能账户 | 可选，取决于合约配置 |
| 多签账户 | 核心特性，必须达到阈值才执行 |

多签的多人确认是合约层强制的，不是靠人际信任——即使某个签名方想单独行动，合约也会拒绝。

### 4. 是否支持恢复、限额或自动化策略

**EOA**：全部不支持。私钥丢了就永久失去访问权，没有任何恢复机制。

**智能账户**：
- **社交恢复**：预设可信地址，私钥丢失后可通过多人投票恢复访问权
- **限额控制**：可设置单笔交易上限、每日转账限额
- **Session Key**：临时授权特定 DApp 在限定条件下操作，时间到自动失效
- **自动化**：可结合链上条件触发（如价格到达某点自动执行）

**多签账户**：
- 支持时间锁（提案通过后延迟 N 小时执行）
- 支持签名方变更（通过多签本身投票增减成员）
- 不支持灵活的单账户恢复，丢失密钥需其他签名方重新配置

### 5. 典型使用场景

**EOA**：
- 个人日常使用，快速交互 DApp
- 测试网操作、小额资产管理
- 不适合存放大额资产

**智能账户**：
- 需要更好用户体验的消费级产品（无 Gas 体验、一键批量交易）
- AI Agent 自主管理资金（Cobo Agentic Wallet 这类）
- 企业级单人账户需要风控策略的场景

**多签账户**：
- DAO 国库管理
- 团队共管资金（如项目方多签）
- 家庭/机构资产托管，防止单点失误或内部作恶

### 6. 主要风险点

**EOA**：
- 私钥/助记词泄露 = 资产归零，无法追回
- 单点故障，没有任何容错
- 钓鱼签名：一个恶意签名可能清空账户

**智能账户**：
- 合约本身可能有漏洞，代码风险叠加在私钥风险之上
- 恢复机制如果设计不当，反而扩大了攻击面（例如恢复人被攻击）
- Session Key 权限设置过宽可能被滥用

**多签账户**：
- 多个签名方同时被社会工程攻击（协同作恶或协同被骗）
- 密钥管理复杂度高，签名方之间协调成本大
- 时间锁期间无法紧急响应（双刃剑）

风险集中度：EOA > 智能账户 > 多签账户
操作灵活性：EOA > 智能账户 > 多签账户
恢复能力：  多签账户 > 智能账户 > EOA
适合资产规模：EOA（小额）/ 智能账户（中）/ 多签（大额/团队）
