## Long-term Memory for AI Agents
### 1.为什么需要外部记忆
LLM 本身无状态。每次调用都是全新的，模型不保留任何上下文。

Agent 的"记忆"= 开发者在每次调用前，把相关历史手动注入 prompt。

就像每次和一个失忆的人对话——你必须在开场白里告诉他"上次我们聊了什么、你是谁、你的目标是什么"，他才能接着干活。没有外部记忆系统，Agent 只能处理单轮任务，无法支撑跨会话的连续工作流。

### 2.四种记忆存储形态

| 类型 | 实现方式 | 适合场景 | 局限 |
|---|---|---|---|
| In-context | 直接拼进 prompt | 短期对话历史 | token 上限，成本高 |
| 向量数据库 | 语义 embedding + 检索 | 大量非结构化历史 | 召回精度依赖 embedding 质量 |
| 结构化数据库 | SQL / KV store | 用户偏好、事实信息 | 不适合模糊查询 |
| Agent 自写摘要 | 每轮结束让 Agent 压缩总结 | 长期会话压缩 | 摘要可能丢失细节 |

#### 应用场景
通常不是单选——生产级 Agent 会组合使用：
- 短期用 in-context
- 长期用向量库语义召回
- 用户画像用结构化存储

### 3.长期一致性的真正挑战
#### 不是"存得越多越好"

记忆注入过多会导致：
- Context 被旧信息稀释，当前任务注意力下降
- 过时信息干扰判断（用户偏好已经改变但旧记录仍在召回）
- Token 成本线性增长

#### 三个核心设计问题

1. **何时召回**：每次都召回 vs 判断相关性再召回
2. **召回多少**：Top-K 相关片段，K 怎么定
3. **记忆有效期**：如何标记/删除过时记忆（用户换工作了，旧的职业信息还要保留吗）

#### 当前主流方案

- **MemGPT / Letta**：Agent 自主管理内外存储，决定什么进 context、什么存档
- **Mem0**：专门做 Agent 记忆层的开源库，自动提取、更新、召回记忆
- **LangGraph + checkpointer**：图结构工作流 + 状态持久化


**记忆系统的难点不是存储，是检索策略和遗忘策略。**
